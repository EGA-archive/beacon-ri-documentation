Querying the API
Beacon PI accepts two types of request methods: GET and POST.

You can find further introduction to Beacon parameters in  Beacon v2 Documentation.

GET Method
GET method is a bit more limited, as this method only accepts request parameters, five generic parameters and ontology filters.

Query Parameters
These are the defined set of parameters (key-value pair) attached to the end of a URL used to provide additional information to a web server when making requests.

Response types
When it comes to Beacon entry type responses, there are two parameters that make a response more or less detailed, or split or not per result set.
These are:
• includeResultsetResponses
• requestedGranularity

The parameter includeResultsetResponses lets you choose how you want the result to be returned, per result set or not, and whether to include only positive results, only negative results, or all of them. Depending on that, you can choose between the following four options:

• HIT: positive (true and/or count > 0) result sets included.
• MISS: negative (false and/or count = 0) result sets included.
• ALL: positive (true and/or count > 0) and negative (false and count = 0) result sets included.
• NONE: only a response summary (true/false and/or count), not split per result set.

The parameter requestedGranularity lets you choose the level of detail you want to be included in the response. Bear in mind that a Beacon can restrict the granularity you will get in the response. You can see what granularity you are finally getting in the meta part of the response, in returnedGranularity. You can choose between the following three types of granularity:

• record: all the detailed documents matching your query are included.
• count: only the number of documents that match your query.
• boolean: only an exists (true/false) property in response to your query.

Note:
The combination NONE + record does not make sense, so the Beacon will not satisfy this query.

POST query examples
Note symbol
You can either make a query with a scope field for each of the filters or just not insert the scope in the filter object. If scope is not stated, then, the scope of the filter will be the same than the endpoint being queried.

Beacon Range Query
curl \
  -H 'Content-Type: application/json' \
  -X POST \
  -d '{
    "meta": { "apiVersion": "2.0" },
    "query": {
      "requestParameters": {"alternateBases": "G", "referenceBases": "A", "referenceName": "22", "start": [16050074], "end": [16050568], "variantType": "SNP", "assemblyId": "GRCh37"},
      "includeResultsetResponses": "HIT",
      "pagination": { "skip": 0, "limit": 10 },
      "testMode": false,
      "requestedGranularity": "record"
    }
  }' \
  http://localhost:5050/api/g_variants

Simple CURIE based filters query
curl \
  -H 'Content-Type: application/json' \

  -d '{

      "pagination": { "skip": 0, "limit": 10 },
      "testMode": false,
      "requestedGranularity": "record"
    }
  }' \
  http://localhost:5050/api/individuals

Note: You are hitting a special alphanumeric corner case. In such cases, we make the id to be the label of the assayCode, and then give the value for it. If we follow the "usual" mechanics, the query wouldn't work, as there is no direct relationship between field and value, because as you have been figuring out, this is an array that can have multiple values, so we need a guide for where to assign the value from the filters thrown in the queries to, and we use assayCode label for that.
